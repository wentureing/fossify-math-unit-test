## 一、项目概述与测试目标

### 1.1 应用简介

Fossify Calculator 是一款完全开源的 Android 计算器应用，采用 GPL v3 协议发布。该应用具备离线运行能力，无需申请任何网络权限，且不收集用户数据。

| 维度 | 描述 |
|------|------|
| **应用名称** | Fossify Calculator |
| **应用类型** | Android 开源计算器 |
| **开源协议** | 完全开源（GPL v3） |
| **离线支持** | ✅ 无需网络权限 |
| **隐私保护** | ✅ 不收集用户数据 |

### 1.2 核心功能

Fossify Calculator 不仅支持基础的算术运算，还具备高级计算能力与单位转换功能，具体功能模块如下：

| 测试模块 | 功能描述 |
|----------|----------|
| **基础运算** | 加减乘除、小数、负数运算 |
| **高级运算** | 幂运算（^）、开方（√）、百分比（%） |
| **单位转换** | 9大类109种单位相互转换 |
| **计算历史** | 历史记录保存、查看、清空 |
| **自定义设置** | 振动开关、屏幕常亮、主题颜色 |
| **桌面小部件** | App Widget 快捷计算 |

### 1.3 测试目标与意义

本次测试从三个维度展开，旨在全面评估应用的质量与可靠性：

| 测试维度 | 目标 | 意义 |
|----------|------|------|
| **功能正确性** | 验证核心计算逻辑准确性 | 保证用户体验基础 |
| **代码质量** | 提升代码可维护性 | 降低后期维护成本 |
| **性能稳定性** | 确保流畅运行体验 | 提升用户满意度 |


## 二、系统架构与代码规模分析

### 2.1 技术栈概览

项目采用纯 Kotlin 开发，混合使用传统 View 系统与 Jetpack Compose，数据持久化采用 Room 数据库，数学表达式解析依赖 EvalEx 库。

| 技术 | 用途 | 版本要求 |
|------|------|----------|
| **Kotlin** | 主要开发语言 | 1.9+ |
| **Android SDK** | 目标平台 | API 33 (Android 13) |
| **Jetpack Compose** | 设置页面 UI | BOM 2024.x |
| **Room Database** | 本地持久化存储 | 2.x |
| **EvalEx** | 数学表达式解析库 | 最新版 |
| **BigDecimal** | 高精度数值计算 | DECIMAL128 |
| **JUnit 4** | 单元测试框架 | 4.13.2 |
| **MockK** | Mock 框架 | 1.13.x |
| **Robolectric** | Android 组件测试 | 4.x |
| **Espresso** | UI 集成测试 | 3.5.x |
| **Jacoco** | 代码覆盖率工具 | 0.8.x |

### 3.2 设计模式识别

| 模式名称 | 应用位置 | 说明 |
|----------|----------|------|
| **MVP 变体** | MainActivity + CalculatorImpl | View-Presenter 分离 |
| **策略模式 (Strategy)** | Converter 接口 + 9种实现 | 单位转换算法封装 |
| **单例模式 (Singleton)** | CalculatorDatabase | 全局唯一数据库实例 |
| **观察者模式 (Observer)** | OnUnitChangedListener | 单位变更通知回调 |
| **工厂模式 (Factory)** | ValueWithUnit 创建 | 统一创建带单位的值 |

### 3.3 核心数据流

用户交互的数据流如下：

```
用户点击按钮
    ↓
MainActivity.numpadClicked()
    ↓
CalculatorImpl 构建表达式
    ↓
EvalEx 解析表达式 → BigDecimal 结果
    ↓
NumberFormatHelper 格式化显示
    ↓
showNewResult() 更新 UI

同时: HistoryHelper → Room Database 持久化历史记录
```

### 10.1 内存泄漏检测与分析

#### 10.1.1 LeakCanary 工具介绍

LeakCanary 是 Square 开源的 Android 内存泄漏检测库，能够在应用运行时自动检测 Activity、Fragment 等组件的内存泄漏问题。当检测到对象未被正常回收时，LeakCanary 会自动触发堆转储（Heap Dump），分析引用链并生成详细的泄漏报告。

| 检测项 | 说明 |
|--------|------|
| **检测目标** | Activity / Fragment 等组件生命周期泄漏 |
| **检测原理** | 弱引用监控 + Heap Dump + 引用链分析 |
| **触发条件** | retained 对象超过阈值（默认5个）自动触发 |
| **输出结果** | 系统通知栏提醒 + 详细泄漏分析报告 |

#### 10.1.2 测试方法与流程

1. **集成 LeakCanary**：在 `build.gradle` 中添加依赖，Application 类中初始化安装。
2. **构造泄漏场景**：打开测试页面，模拟 Activity 持有外部引用，关闭页面后制造内存泄漏。
3. **触发检测**：LeakCanary 自动监控对象回收状态，当 retained 对象达阈值时触发 Heap Dump。
4. **分析结果**：通过系统通知栏获取泄漏提醒，查看引用链定位泄漏根因。

### 10.4 历史记录数字格式缺陷分析

在系统功能测试过程中，发现历史记录模块存在与系统 Locale 切换相关的数字格式解析缺陷。该缺陷涉及数据存储、界面显示和结果加载三个环节，当用户在计算完成后切换系统语言/地区设置，再次访问历史记录时会出现数字显示异常甚至计算结果错误。

#### 10.4.1 缺陷触发流程

| 步骤 | 用户操作 | 系统状态 | 关键行为 |
|------|----------|----------|----------|
| **Step 1** | 计算并存储结果 | 系统 Locale 为美国（en_US） | 计算器执行运算后，调用 `result.format()` 将结果格式化为带千位分隔符和小数点的字符串（如 `"1,234.56"`），并写入数据库历史记录表 |
| **Step 2** | 更改系统 Locale | 切换为德国（de_DE） | 系统小数点变为逗号 `,`，千位分隔符变为点 `.`。数据库中已存储的历史数据保持不变，仍为美式格式字符串 |
| **Step 3** | 查看历史记录 | 在德国 Locale 环境下 | `HistoryAdapter` 直接显示数据库中的原字符串 `"1,234.56"`，但德国用户期望看到 `"1.234,56"`（千位符为点，小数点为逗号），显示不符合当前 Locale 习惯 |
| **Step 4** | 点击历史条目加载 | 在德国 Locale 环境下 | `CalculatorImpl` 将历史字符串 `"1,234.56"` 传入 `addNumberToFormula`，随后 `addThousandsDelimiter` 尝试按当前 Locale 重新格式化。由于 `numbersRegex` 和 `NumberFormatHelper` 均基于德国 Locale 配置，美式格式的逗号和点被错误解析，导致数值失真 |

#### 10.4.2 各环节缺陷根因

| 环节 | 涉及文件与方法 | 缺陷描述 |
|------|----------------|----------|
| **存储** | `CalculatorImpl.kt` — `calculateResult` | 调用 `result.format()` 存储 Locale 敏感字符串（如 `"1,234.56"`），而非标准化数值。数据库中保留了特定 Locale 的格式化结果 |
| **显示** | `HistoryAdapter.kt` — `bindView` | 直接显示数据库原字符串，未根据当前系统 Locale 重新格式化，导致跨 Locale 场景下显示不符合用户习惯 |
| **加载** | `CalculatorImpl.kt` — `addNumberToFormula` / `addThousandsDelimiter` | 将历史字符串当作当前 Locale 格式处理，依赖 `NumberFormatHelper.formatForDisplay` 解析。旧格式（美式）与新 Locale（德式）不兼容，逗号和点的语义冲突导致数值被错误拆分，最终加载到计算器的数字发生精度丢失或数值错误 |

#### 10.4.3 数值解析错误示例（德国 Locale 场景）

以存储值 `"1,234.56"` 在德国 Locale 下加载为例：

1. `removeGroupingSeparator` 先按德国 Locale 处理：将点 `.` 当作千位分隔符移除，字符串变为 `"1,23456"`；再将逗号 `,` 替换为点，得到 `"1.23456"`。
2. `"1.23456"` 转为 `BigDecimal` 后数值变为 `1.23456`，原始值 `1234.56` 的整数部分被完全破坏。
3. 后续重新格式化的结果进一步偏离预期，用户看到的数字与原始计算结果严重不符。

#### 10.4.4 修复方向

| 层面 | 建议措施 | 预期效果 |
|------|----------|----------|
| **数据存储** | 数据库统一存储标准化数字字符串（如 `"1234.56"`），不带千位分隔符，小数点固定为点 `.` | 消除 Locale 敏感性，保证数据持久层格式稳定 |
| **界面显示** | 读取历史记录时，通过 `NumberFormatHelper` 根据当前系统 Locale 动态格式化后再显示 | 无论用户切换至何种 Locale，历史结果均按当地习惯正确呈现 |
| **结果加载** | 加载历史条目时直接使用标准化数字传入计算引擎，避免再次经过 Locale 敏感的解析流程 | 杜绝数值解析错误，保证加载后的计算结果与原始结果一致 |

| **Android 依赖隔离困难** | 使用 MockK mock 构造函数和 relaxed mock | Mock 策略需精心设计，避免过度 mock 导致测试失去意义 |
| **Robolectric 与真实设备差异** | 关键场景同时编写 Robolectric + Espresso 双版本 | 单元测试快速验证逻辑，集成测试验证真实行为 |
| **Jacoco 覆盖率合并问题** | 分别生成单元/集成报告，手动交叉验证 | 注意排除自动生成代码，关注业务逻辑覆盖率 |
| **单例模式的测试污染** | 使用独立测试数据库文件 + tearDown 清理 | 测试隔离至关重要，避免跨测试状态泄漏 |
| **温度转换的非线性特性** | 在 Roundtrip 测试中过滤掉 TemperatureConverter | 特殊边界条件需要单独处理，不能一刀切 |
| **UI 测试的不稳定性** | 使用 IdleResource + 显式等待 | 避免 sleep 硬编码，优先使用同步机制 |
